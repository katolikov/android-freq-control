# FreqControl

A small C++17 library for pinning CPU / GPU / DRAM frequencies and CPU
thread affinity on Android devices. Built for Samsung Exynos but
extensible to any Linux SoC that exposes `cpufreq` / `devfreq` through
sysfs.

Per-device configuration is auto-generated by probing the device over
`adb` — there is no hand-written JSON, no per-device branching in C++,
and no runtime parsing. Adding a new SoC is one command.

---

## What it gives you

```cpp
#include "freq_control/freq_controller.h"
#include "freq_control/thread_affinity.h"

using namespace freq_control;

FreqController ctl;
ctl.SetFrequencyMode(FrequencyMode::kBoost);   // or SetFrequencyCustomId(1001)
ctl.SetClocks();                                // save current, apply profile
SetThreadAffinity(CpuCluster::kPrime);          // pin caller to prime cores
// ... hot work ...
ResetThreadAffinity();
ctl.UnsetClocks();                              // restore saved freqs
```

The public surface is four `FreqController` methods plus two free
functions:

| API                                          | Implementation                          |
| -------------------------------------------- | --------------------------------------- |
| `FreqController::SetFrequencyMode(mode)`     | picks one of 5 named profiles           |
| `FreqController::SetFrequencyCustomId(id)`   | picks a custom workload profile         |
| `FreqController::SetClocks()`                | sysfs writes, save/apply with rollback  |
| `FreqController::UnsetClocks()`              | sysfs writes, restore saved values      |
| `SetThreadAffinity(cluster, tid=0)`          | `sched_setaffinity(2)` syscall          |
| `ResetThreadAffinity(tid=0)`                 | `sched_setaffinity(2)` syscall          |

Five frequency modes (in `FrequencyMode`): `kPowerSave`, `kBalanced`,
`kPerformance`, `kBoost`, `kMaximum`. Four CPU clusters (in
`CpuCluster`): `kLittle`, `kMid`, `kBig`, `kPrime`.

## Architecture

```
include/freq_control/      public headers (API surface)
src/                       implementation
config/soc/<soc>.h         GENERATED per-SoC tables (one file per device)
tools/gen_device_config.py adb-probes a device, emits config/soc/<soc>.h
app/freq_ctl.cc            verification CLI
```

Each per-SoC header declares `constexpr` tables: rail sysfs paths,
per-mode (min, max) targets per rail, and CPU-cluster → CPU-id maps.
`src/device_config.cc` selects which header to compile against via
`-DFREQ_CONTROL_SOC=<name>`. Switching SoCs is a recompile.

Frequency writes are ordered to never violate the kernel's `min <= max`
invariant during transition; the controller's destructor auto-rolls
back if `SetClocks()` was called and `UnsetClocks()` was not.

The thread-affinity API issues `syscall(__NR_sched_setaffinity, ...)`
directly rather than the libc wrapper, so it links cleanly against
bionic without pulling extra symbols.

## Adding a new device (one command)

```bash
adb devices
# -> R5CY71BJJ9D    device

python3 tools/gen_device_config.py \
    --adb-serial R5CY71BJJ9D \
    -o config/soc/s5e9955.h
```

The script reads `ro.soc.model`, walks `/sys/devices/system/cpu/cpufreq/policy*`
to learn CPU topology, finds the GPU and MIF nodes under
`/sys/class/devfreq/`, and synthesises the five modes by index-picking
percentiles from each rail's `scaling_available_frequencies`. Every
emitted value is therefore guaranteed valid on that kernel.

The script handles 1–4 CPU clusters automatically. Extend it for
unusual SoC topologies or to add `custom` workload profiles.

## Build

Requires Android NDK (r25+, r27 used in development).

```bash
NDK=$ANDROID_HOME/ndk/27.0.12077973
cmake -S . -B build \
    -DCMAKE_TOOLCHAIN_FILE=$NDK/build/cmake/android.toolchain.cmake \
    -DANDROID_ABI=arm64-v8a \
    -DANDROID_PLATFORM=android-30 \
    -DFREQ_CONTROL_SOC=s5e9955
cmake --build build
```

Outputs:
- `build/libfreq_control.a` — link this into your binary.
- `build/freq_ctl` — verification CLI.

Build flags `-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Werror`
are enforced by `CMakeLists.txt`.

## Verify on device

```bash
adb push build/freq_ctl /data/local/tmp/
adb shell chmod +x /data/local/tmp/freq_ctl

# Read-only checks (no root required):
adb shell /data/local/tmp/freq_ctl probe         # all paths exist & parse
adb shell /data/local/tmp/freq_ctl list          # dump all profiles
adb shell /data/local/tmp/freq_ctl affinity big  # syscall path works

# Write checks (need root / userdebug build):
adb shell /data/local/tmp/freq_ctl set performance
adb shell /data/local/tmp/freq_ctl cycle boost 5
```

## Notes

- Frequency writes need root or a process with write access to the
  relevant sysfs nodes (typically `system` uid for cpufreq, `root` for
  GPU). The library reports per-path errors cleanly when permissions
  are insufficient.
- Thread affinity does **not** require root for self-affinity.
- This is not a kernel module and does not bypass any kernel checks.
  Requested frequencies are constraints; the kernel still chooses
  within the `[min, max]` window via the active governor.
