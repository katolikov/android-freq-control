# FreqControl

A small C++17 library for pinning CPU / GPU / DRAM frequencies and CPU
thread affinity on Android devices. Built for Samsung Exynos but
extensible to any Linux SoC that exposes `cpufreq` / `devfreq` through
sysfs.

Per-device configuration is auto-generated by probing the device over
`adb` — there is no hand-written JSON and no runtime parsing. Every
supported SoC is compiled into the binary and picked at runtime via a
`Device` enum that the generator extends each time a new device is added.

---

## What it gives you

```cpp
#include "freq_control/freq_controller.h"
#include "freq_control/thread_affinity.h"

using namespace freq_control;

// One-time at startup: pick the device (or read ro.soc.model and use
// SetActiveDeviceByName(...)). Optional in single-device builds -- the
// first registered device is the default.
SetActiveDevice(Device::kS5E9955);

FreqController ctl;
ctl.SetFrequencyMode(FrequencyMode::kBoost);   // or SetFrequencyCustomId(1001)
ctl.SetClocks();                                // save current, apply profile
SetThreadAffinity(CpuCluster::kPrime);          // pin caller to prime cores
// ... hot work ...
ResetThreadAffinity();
ctl.UnsetClocks();                              // restore saved freqs
```

The public surface is four `FreqController` methods plus two free
affinity functions plus the runtime device-selection API:

| API                                          | Implementation                          |
| -------------------------------------------- | --------------------------------------- |
| `SetActiveDevice(Device)`                    | selects which compiled-in SoC is active |
| `SetActiveDeviceByName(const char*)`         | same, by SoC name (e.g. `"s5e9955"`)    |
| `GetActiveDevice()` / `GetDeviceConfig()`    | inspect current selection               |
| `FreqController::SetFrequencyMode(mode)`     | picks one of 5 named profiles           |
| `FreqController::SetFrequencyCustomId(id)`   | picks a custom workload profile         |
| `FreqController::SetClocks()`                | sysfs writes, save/apply with rollback  |
| `FreqController::UnsetClocks()`              | sysfs writes, restore saved values      |
| `SetThreadAffinity(cluster, tid=0)`          | `sched_setaffinity(2)` syscall          |
| `ResetThreadAffinity(tid=0)`                 | `sched_setaffinity(2)` syscall          |

Three frequency modes (in `FrequencyMode`):

- `kPowerSave` — clocks held low; minimal extras.
- `kPerformance` — clocks pinned at the top of each rail; tunable extras
  applied at moderate values.
- `kBoost` — same rails as `kPerformance`; tunable extras pushed harder.

Four CPU clusters (in `CpuCluster`): `kLittle`, `kMid`, `kBig`, `kPrime`.

### Per-mode tunables

In addition to CPU/GPU/devfreq rails, each profile carries a list of
`Tunable { name, sysfs_path, value }` knobs that can be:

- **sysfs-bound** — when `sysfs_path` is non-null the integer `value` is
  written to that node during `SetClocks()` and restored on `UnsetClocks()`
  alongside the rails. Use this for vendor `MINLOCK`-style nodes.
- **application-readable** — when `sysfs_path` is null the value is
  exposed to your code via `ctl.GetTunable("NAME", fallback)` so you can
  apply it however you like (e.g. feed a CPU mask to `sched_setaffinity`).

The generator emits sensible defaults for `SCHED_COMMON_TARGET_LATENCY`
and `MEM_MO_SCEN_ID_CUSTOMIZED`. Extend `_build_tunables` in
`tools/gen_device_config.py` to add more, or hand-edit the generated
`include/freq_control/soc/<soc>.h`.

## Architecture

```
include/freq_control/
  device_config.h          static types + runtime device-selection API
  device_registry.h        GENERATED: Device enum + registry table
  soc/<name>.h             GENERATED: per-SoC freq tables (one per device)
  freq_controller.h
  thread_affinity.h
src/                       implementation
tools/gen_device_config.py adb-probes a device; emits soc/<name>.h and
                           re-emits device_registry.h
app/freq_ctl.cc            verification CLI
```

Each per-SoC header declares `constexpr` tables: rail sysfs paths,
per-mode (min, max) targets per rail, and CPU-cluster → CPU-id maps.
`device_registry.h` includes every `soc/<name>.h` and exposes them
through a single `Device` enum + lookup table. `SetActiveDevice` picks
which one the runtime uses; switching SoCs is a runtime call, not a
recompile.

Frequency writes are ordered to never violate the kernel's `min <= max`
invariant during transition; the controller's destructor auto-rolls
back if `SetClocks()` was called and `UnsetClocks()` was not.

The thread-affinity API issues `syscall(__NR_sched_setaffinity, ...)`
directly rather than the libc wrapper, so it links cleanly against
bionic without pulling extra symbols.

## Adding a new device (one command)

```bash
adb devices
# -> R5CY71BJJ9D    device

python3 tools/gen_device_config.py --adb-serial R5CY71BJJ9D
```

The script:

1. Reads `ro.soc.model`, walks `/sys/devices/system/cpu/cpufreq/policy*`
   to learn CPU topology, finds the GPU and MIF nodes under
   `/sys/class/devfreq/`, and synthesises the five modes by index-picking
   percentiles from each rail's `scaling_available_frequencies`. Every
   emitted value is guaranteed valid on that kernel.
2. Writes `include/freq_control/soc/<soc>.h`.
3. Rescans `include/freq_control/soc/` and rewrites
   `include/freq_control/device_registry.h` so the new SoC is appended
   to the runtime `Device` enum.

Rebuild and the new device is selectable via
`Device::k<SOC>` or `SetActiveDeviceByName("<soc>")`.

The script handles 1–4 CPU clusters automatically. Extend it for
unusual SoC topologies or to add `custom` workload profiles.

## Build

Requires Android NDK (r25+, r27 used in development).

```bash
export ANDROID_NDK=$HOME/Library/Android/sdk/ndk/27.0.12077973
./build.sh
```

`build.sh` picks the NDK from `$ANDROID_NDK`, then `$ANDROID_NDK_HOME`,
then the highest-versioned directory under `$ANDROID_HOME/ndk/`. Options:

```
./build.sh --soc s5e9955 --abi arm64-v8a --api 30 --clean -j 8
```

Direct CMake also works:

```bash
cmake -S . -B build \
    -DCMAKE_TOOLCHAIN_FILE=$ANDROID_NDK/build/cmake/android.toolchain.cmake \
    -DANDROID_ABI=arm64-v8a \
    -DANDROID_PLATFORM=android-30 \
    -DFREQ_CONTROL_SOC=s5e9955
cmake --build build
```

Outputs:
- `build/libfreq_control.a` — link this into your binary.
- `build/freq_ctl` — verification CLI.

Build flags `-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Werror`
are enforced by `CMakeLists.txt`.

## Verify on device

Quick end-to-end with `run.py` (prints BEFORE / DURING / AFTER clocks
and verifies that `UnsetClocks()` restored the original values):

```bash
./run.py -s <SERIAL> -m boost           # hold 5 s
./run.py -s <SERIAL> -m performance -d 10
./run.py -s <SERIAL> -m 1001            # by custom id
```

Or call the binary directly:

```bash
adb push build/freq_ctl /data/local/tmp/
adb shell chmod +x /data/local/tmp/freq_ctl

# Read-only checks (no root required):
adb shell /data/local/tmp/freq_ctl probe         # all paths exist & parse
adb shell /data/local/tmp/freq_ctl list          # dump all profiles
adb shell /data/local/tmp/freq_ctl affinity big  # syscall path works

# Write checks (need root / userdebug build):
adb shell /data/local/tmp/freq_ctl set performance
adb shell /data/local/tmp/freq_ctl cycle boost 5
```

## Notes

- Frequency writes need root or a process with write access to the
  relevant sysfs nodes (typically `system` uid for cpufreq, `root` for
  GPU). The library reports per-path errors cleanly when permissions
  are insufficient.
- Thread affinity does **not** require root for self-affinity.
- This is not a kernel module and does not bypass any kernel checks.
  Requested frequencies are constraints; the kernel still chooses
  within the `[min, max]` window via the active governor.
